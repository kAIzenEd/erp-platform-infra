# PROD compose — ETS ERPNext

This folder is the **production compose stack** for ETS/ACA ERPNext.

| File | Purpose |
|---|---|
| [`compose.yml`](./compose.yml) | Full ERPNext stack (db, redis, configurator, create-site, backend, frontend, websocket, queues, scheduler). Mirrors the dev stack image-for-image so backups round-trip cleanly. |
| [`.env.example`](./.env.example) | Variables the compose references. **Copy to `.env` and fill in real values**; `.env` is gitignored. |
| `.env` | Real secrets for this host. Never committed. Generated by `scripts/prod/bootstrap-host.sh`. |

## Quick start

Assumes a Linux host with Docker + Compose plugin installed, and the user can run `docker` without `sudo`.

```bash
# One-time host bootstrap (creates /opt/aca, clones custom apps, writes .env)
sudo mkdir -p /opt/aca && sudo chown $USER:$USER /opt/aca
cd /opt/aca
git clone https://github.com/kAIzenEd/erp-platform-infra.git infra
/opt/aca/infra/scripts/prod/bootstrap-host.sh

# Bring the stack up (first boot ~3-5 min while configurator + create-site run)
cd /opt/aca/infra/compose/prod
docker compose --env-file .env up -d

# Watch the one-shot create-site finish
docker compose --env-file .env logs -f create-site
```

When `create-site` exits with code 0, the site exists. Open `http://<server-ip>:8080/` and log in as `Administrator` with the `ADMIN_PASSWORD` from `.env`.

## Day-2 operations

All under [`../../scripts/prod/`](../../scripts/prod):

| Command | What it does |
|---|---|
| `./scripts/prod/deploy.sh` | `git pull` custom apps, `bench migrate`, clear cache, restart long-running services |
| `./scripts/prod/backup.sh` | `bench backup --with-files` + copy archives out to `./backups/<timestamp>/` |
| `./scripts/prod/restore.sh backups/<dir>` | Restore a backup folder onto this stack (destructive — prompts to confirm) |

## What gets persisted

| Volume | Contents | Backed up by |
|---|---|---|
| `${STACK_NAME}_db_data` | MariaDB data files | `backup.sh` (via `bench backup`) |
| `${STACK_NAME}_sites` | `sites/` tree: site config, attached files, search index | `backup.sh` (via `bench backup --with-files`) |
| `${STACK_NAME}_logs` | Frappe logs (rotation handled by Frappe) | Not backed up — disposable |
| `${STACK_NAME}_redis_queue_data` | Persistent queue (so deferred jobs survive a restart) | Not backed up — disposable |

## What lives outside the volumes (bind-mounted from the host)

| Host path | Container path | Purpose |
|---|---|---|
| `${REPOS_ROOT}/ets_admissions` | `/home/frappe/frappe-bench/apps/ets_admissions` | Custom admissions app. Update via `git pull` on host, then `./scripts/prod/deploy.sh`. |
| `${REPOS_ROOT}/ets_students` | `/home/frappe/frappe-bench/apps/ets_students` | Custom students app. Same flow. |

## Migration to another host

The procedure is identical for any direction (Mac → Linux server, server A → server B, etc.):

1. **Source host:** `./scripts/prod/backup.sh --label cutover-pre`
2. **Copy:** `rsync -avz backups/cutover-pre/ user@new-host:/opt/aca/infra/backups/cutover-pre/`
3. **New host:** Run `scripts/prod/bootstrap-host.sh` (sets up Docker, clones repos, writes a fresh `.env`)
4. **New host:** `cd compose/prod && docker compose --env-file .env up -d` (creates an empty site first)
5. **New host:** `./scripts/prod/restore.sh backups/cutover-pre` (replaces the empty site with the source data)

See [`docs/runbooks/prod-setup.md`](../../docs/runbooks/prod-setup.md) for the full end-to-end runbook.
